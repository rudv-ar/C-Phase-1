## What `&a` actually means in memory terms

`a` is a **local variable** inside `main`. From our memory table, that means it lives on the **Stack** at `rw-`.

When you write `&a`, you're saying: **"give me the memory address of `a` on the stack."**

`scanf`'s job is:
1. Read your input from keyboard
2. Parse it as an integer
3. **Write that integer to the address you gave it**

So `scanf("%d", &a)` means: *"parse an int and write it to this stack address."* That's legal — stack is `rw-`.

---

## What happens when `&a` is missing

```
scanf("%d");   // you told scanf WHAT to read, but not WHERE to write it
```

scanf still does its job — it reads your input, parses the integer. But now it has **no address to write to**. It doesn't crash here yet. It goes looking for where the address *should* be.

In x86-64, function arguments are passed via **registers** (`rdi`, `rsi`, `rdx`...). The calling convention says:
- `rdi` = 1st argument = `"%d"` (the format string, from `.rodata`)
- `rsi` = 2nd argument = **should be `&a`, but you never set it**

Since you never set `rsi`, it contains **whatever garbage value was left there from a previous operation** — completely random. Could be `0x0`, could be `0x7fff...`, could be anything.

scanf then does: *"I'll write the integer to whatever address is in `rsi`"* — the garbage address.

---

## Why SIGSEGV — Using the Memory Table

```
Address         Region          rwx     What happens
─────────────────────────────────────────────────────────────────
0x0000          Unmapped        ---     ← instant SIGSEGV
0x1000          .text (code)    r-x     ← SIGSEGV (no w)
0x2000          .rodata         r--     ← SIGSEGV (no w)
0x3dd0          .dynamic/.got   rw-     ← might survive (rare luck)
0x4020          .bss            rw-     ← might survive (rare luck)
[stack]         Stack           rw-     ← would be fine IF it landed here
[garbage]       UNMAPPED        ---     ← almost certainly this → SIGSEGV
```

The garbage value in `rsi` is almost certainly pointing at an **unmapped region** — memory that doesn't belong to your process at all. The CPU checks permissions before every write. When it sees the target address has `---` (not mapped / no permissions), it raises a **hardware exception** instantly. The kernel catches that exception and sends your process **SIGSEGV** — "you touched memory you're not allowed to."

Your fish shell then printed: `terminated by signal SIGSEGV (Address boundary error)`.

## Why the behavior differed across inputs — this is the key insight

From your error file:

```
Input: "234"      → SIGSEGV        ✗
Input: "sf"       → prints 0, OK   ✓
Input: "234.234"  → SIGSEGV        ✗
Input: huge num   → SIGSEGV        ✗
```

This is **not random**. Here's exactly why each behaved differently:

### `"sf"` — No crash
scanf tried to parse `"sf"` as `%d` (integer). `s` is not a digit. scanf **fails to parse immediately**, returns without writing anything at all. It never touches the garbage address. `a` stays uninitialized, printf prints `0` (garbage that happened to be 0 on the stack). No write ever happened → no SIGSEGV.

### `"234"` → SIGSEGV
scanf successfully parses `234`. Now it **has a value to write**. It tries to write `234` to the garbage address in `rsi`. CPU checks permissions → `---` → SIGSEGV.

### `"234.234"` → SIGSEGV
scanf parses `234` successfully (stops at the `.`). Same as above — has a value, tries to write to garbage address → SIGSEGV.

### Huge number → SIGSEGV
scanf parses some value. Same thing — write attempt to garbage address → SIGSEGV.

## The one-line summary

`scanf` needed a **writable stack address** (`rw-`) to store your input. You gave it nothing, so it used garbage. Garbage pointed to unmapped memory (`---`). The CPU won't let anyone write there. SIGSEGV.

The correct `&a` would have given it a proper stack address — same `rw-` region, completely valid, no crash...
